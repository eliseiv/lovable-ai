# ADR-004 — Adapty как источник истины (вебхуки + getProfile vs только вебхуки)

**Статус:** Accepted · **Дата:** 2026-06-02

## Context

Биллинг — мобильные подписки/IAP через Adapty. iOS ведёт покупку через Adapty SDK; backend гейтит квоты генерации по тарифу. Вопрос: на чём backend основывает решение о правах доступа.
- Только вебхуки: быстро, но вебхук может быть потерян/задержан → расхождение прав.
- Только `getProfile` на каждом гейте: всегда консистентно, но добавляет latency и зависимость от доступности Adapty на горячем пути.

## Decision

**Adapty — источник истины по подпискам. Backend использует вебхуки + периодический `getProfile`-ресинк, локальный `subscriptions` — кэш.**

- Вебхуки (`POST /billing/webhook/adapty`) — основной канал апдейта `subscriptions`; обработка **идемпотентна** по `adapty_event_id` (таблица `billing_events`), верификация секрета Adapty.
- На гейте (`POST /projects`, `/edits`) по умолчанию читаем локальный кэш `subscriptions` (низкая latency); политика «всегда ли дополнительно дёргать `getProfile`» и частота фоновой ресинхронизации — открытый вопрос [Q-BILLING-2](../99-open-questions.md#q-billing-2).
- Периодический `getProfile`-ресинк (Celery beat) подстраховывает от пропущенных вебхуков.
- Бизнес-квота (`plan_quotas.monthly_generations`) и технический cost-cap Claude (`job_budget_usd`) — **две независимые величины**.

## Consequences

**Плюсы:** низкая latency на гейте; устойчивость к потерянным вебхукам через ресинк; идемпотентность исключает двойную обработку; чёткое разделение бизнес-квоты и себестоимости.
**Минусы:** окно рассинхронизации между вебхуком и ресинком (mitigation — частота ресинка / опц. `getProfile` на гейте, [Q-BILLING-2]); зависимость от доступности Adapty для актуальности прав.

## Alternatives

- **Только вебхуки.** Отвергнут: потеря/задержка вебхука → неверные права без механизма самокоррекции.
- **Только `getProfile` на каждом гейте.** Отвергнут как дефолт: latency и зависимость от Adapty на горячем пути генерации. Остаётся опцией для критичных проверок ([Q-BILLING-2]).
- **Собственный учёт чеков/IAP.** Отвергнут: дублирование того, что Adapty уже делает; сложнее и рискованнее.
