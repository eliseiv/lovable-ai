# ADR-028 — Детерминированный детект языка из ИСХОДНОГО промпта + серверная инжекция директивы (ревизия ADR-025)

**Статус:** Accepted
**Дата:** 2026-06-11
**Ревизует:** [ADR-025](ADR-025-content-language-autodetect-spec-marker.md) (механизм определения языка контента — заменяется; маркер `**Content language:**` и его роль downstream — сохраняются)
**Связано:** [ADR-020](ADR-020-agent-structured-output-tool-use-tolerant-parse-retry.md) (output-схемы §I.1a), [pipeline §03 → Язык/локализация](../modules/pipeline/03-architecture.md#языклокализация-контента-сайта--детерминированный-детект-adr-028-ревизует-adr-025), [08-product-decisions §Локализация](../08-product-decisions.md#локализация--язык-сгенерированного-сайта--язык-пользователя-авто-детект-adr-025-ревизия-adr-028), [03-data-model → generation_jobs](../03-data-model.md#generation_jobs), [Q-LOCALE-1](../99-open-questions.md#q-locale-1)

## Context

ADR-025 локализовал сайт «на языке пользователя» через **мягкую промпт-инструкцию**: Agent 1 (Interviewer) задаёт вопросы «на языке промпта»; Agent 2 (Spec writer) **сам** детектит язык из всего ввода (промпт + Q&A) и ставит маркер `**Content language:**`. Это **не детерминированно** — Sonnet (Agent 1, [ADR-023 R1](ADR-023-agent3-token-budget-thinking-room.md)) соблюдает инструкцию не на 100 %.

**Прод-баг (воспроизведён):** один и тот же **русский** промпт даёт то русские вопросы (норма), то английские (баг) при одном и том же коде. Каскад отказа:

1. Agent 1 — **первый** агент в пайплайне, у него нет якоря языка, кроме самого промпта; при сбое инструкции он выдаёт **английские** вопросы на русский промпт.
2. Пользователь отвечает на вопросы **по-английски** (следует языку вопросов).
3. Agent 2 по приоритету ADR-025 (**ответы > промпт**) детектит «английский» → весь сайт английский, хотя **исходный** промпт был русский.

**Корень:** язык определяется **недетерминированно** (само-детект LLM на каждом шаге) и **поздно** (Agent 2), тогда как у Agent 1 — первого user-facing агента — якоря нет. Приоритет ответов усиливает каскад: ошибка Agent 1 «отравляет» ответы, и Agent 2 закрепляет неверный язык.

**Цель ревизии:** язык определяется **один раз, детерминированно, на сервере, из ИСХОДНОГО промпта пользователя** (`project.prompt`, тело `POST /projects`) — до запуска Agent 1 — и **жёстко инжектируется** во весь пайплайн как явная **серверная директива** агентам, а не передетектируется каждым агентом.

## Decision

### 1. Механизм детекта — собственная узкая script-эвристика (без новой зависимости)

Язык исходного промпта определяется **детерминированной серверной функцией** по доминирующему Unicode-**script** текста промпта (не по словарю/n-граммам):

- Подсчитывается доля символов-букв (исключая пробелы, цифры, пунктуацию, emoji) по script-группам: **Cyrillic**, **Latin**, и пр.
- Доминирующий script → язык-директива по фиксированной таблице: **Cyrillic → Russian (ru)**, **Latin → English (en)**. Таблица расширяема, но MVP покрывает ru/en (фактические языки пользователей продукта).
- Результат — пара `(<название языка>, <BCP-47>)`, например `Russian (ru)` или `English (en)`.

**Обоснование выбора script-эвристики vs библиотека `langdetect`/`py3langid`:**
- Язык влияет **лишь на директиву** агентам; **финальную лингвистическую точность даёт LLM** (модель пишет грамотный контент на указанном языке). Поэтому грубого script-детекта **достаточно**.
- Детерминизм здесь — **главное требование**; script-подсчёт детерминирован тривиально и не зависит от версий модели весов/словарей.
- `langdetect`/`py3langid` **шумят на коротких текстах** (типичный промпт — одна-две фразы), могут вернуть `sk`/`uk`/`bg` на короткой кириллице — это привнесло бы новый класс недетерминизма и edge-кейсов ради точности, которая для **директивы** не нужна.
- **Нулевая новая зависимость** — функция чистый Python (Unicode-категории stdlib). `docs/02-tech-stack.md` правки **не требует**.

> **Граница точности (намеренная):** script ≠ язык (украинский/болгарский тоже кириллица; немецкий/французский — латиница). Для **директивы** это приемлемо: директива задаёт **скрипт/семейство**, а конкретный язык в рамках скрипта модель берёт из самого текста промпта, который ей подаётся целиком. Полную многоязычность (различение языков внутри одного script) даёт LLM, не серверный детект. Если в будущем потребуется различать языки внутри script — отдельный ADR + библиотека в `02-tech-stack.md`.

### 2. Где определяется и как хранится — поле `content_language` в `generation_jobs` + миграция

Детект выполняется **один раз на старте фазы interview** (Celery `task_interview` → `_interview`, `app/workers/tasks.py`), до вызова Agent 1, по `project.prompt`. Результат сохраняется в **новое поле** `generation_jobs.content_language` (BCP-47 строка, напр. `ru`/`en`).

**Обоснование явного поля БД (пересмотр запрета ADR-025):** ADR-025 намеренно избегал поля БД и параметра в сигнатурах. Но при детерминированном детекте из исходного промпта появляются два новых требования, которых у ADR-025 не было:
- **Crash-resume:** после рестарта воркер подхватывает джобу в активном state (`INTERVIEWING`/`SPECCING`, диспетчер по `state`); язык должен восстанавливаться **без передетекта** (а исходный промпт к фазе SPECCING уже не единственный сигнал — нужен зафиксированный якорь). Поле в БД — единственный crash-устойчивый канал, переживающий рестарт между фазами.
- **Единый якорь для всех агентов:** Agent 1 запускается **до** Agent 2 и до появления `spec_markdown`; маркер в спеке (канал ADR-025) к моменту Agent 1 ещё не существует. Поле читается обоими агентами из БД.

Маркер `**Content language:**` в `spec_markdown` **сохраняется** (downstream Agent 3/4 читают язык из спеки, как в ADR-025) — но его **значение** теперь приходит из `content_language` (детерминированный серверный детект), инжектируется сервером в директиву Agent 2, а не выбирается моделью.

### 3. Прокидка в агентов — серверная инжекция директивы

- **Agent 1 (Interviewer)** получает инжектируемую **сервером** директиву вида `Generate all questions in <language name> (<bcp-47>).` Язык — из `content_language`, **не** само-детект.
- **Agent 2 (Spec writer)** получает ту же серверную директиву `Generate all user-facing content in <language name> (<bcp-47>).` и **обязан** начать `spec_markdown` маркером `**Content language:** <language name> (<bcp-47>)` со **значением из директивы** (детерминированное серверное значение), а **не** из собственного детекта.
- **Agent 3 (Builder)** и **Agent 4 (Fixer/Editor)** — без изменений относительно ADR-025: читают язык из маркера в спеке (`<html lang>` + контент; сохранение языка). Маркер теперь гарантированно детерминирован.

### 4. Приоритет — ИСХОДНЫЙ промпт (не ответы)

Приоритет языка пересматривается: язык определяется **из исходного промпта пользователя** (`project.prompt`) — это исходное намерение пользователя. **Приоритет ответов отменяется полностью.** Именно приоритет ответов над промптом порождал каскад (ошибочные английские вопросы → английские ответы → английский сайт на русский промпт). При детерминированном детекте из промпта **до** Agent 1 ответы пользователя уже будут на правильном языке (вопросы детерминированно на языке промпта), поэтому отдельный сигнал «ответы» не нужен и вреден.

> **Единый нормативный источник правила приоритета** — [pipeline §03 → Язык/локализация](../modules/pipeline/03-architecture.md#языклокализация-контента-сайта--детерминированный-детект-adr-028-ревизует-adr-025). ADR-025, 08-L-2/L-3, таблица агентов §03 — **ссылаются** на него, не переформулируют. После этой ревизии формулировки «приоритет ответов» / «детект — Agent 2 само-детект» в docs **не остаётся нигде**.

### 5. Fallback — детерминированное правило при неуверенном/смешанном script

Если доминирующий script **не определяется уверенно** (нет буквенных символов — только цифры/emoji/пунктуация; либо ни один script не набирает строгого большинства > 50 % буквенных символов — смешанный ввод):

- **Дефолт — `English (en)`.** Детерминированно, без обращения к модели.

Правило строгое и воспроизводимое: при равной/неоднозначной доле script-ов всегда `en`. Это убирает любой источник недетерминизма в граничных случаях.

## Consequences

**Плюсы.**
- **Детерминизм:** один и тот же промпт → один и тот же язык вопросов/контента **на каждом прогоне** (script-подсчёт не флапает). Корень прод-бага закрыт.
- **Ранний якорь:** язык фиксирован **до** Agent 1 — первый user-facing агент больше не «угадывает».
- **Crash-resume:** язык переживает рестарт воркера (поле в БД), не передетектится.
- **Нулевая новая зависимость:** script-детект — чистый Python.
- **Downstream-канал сохранён:** Agent 3/4 по-прежнему читают язык из маркера в спеке — их контракт не меняется.

**Минусы / риски.**
- **Новое поле + миграция** (revises head `20260608_0001`) — отступление от «без миграции» ADR-025; оправдано crash-resume и ранним якорем (см. Decision §2).
- **script ≠ язык:** различение языков внутри одного script (uk vs ru, de vs en) серверный детект не делает — отдано LLM (директива задаёт script-семейство, текст промпта подаётся модели целиком). Приемлемо для MVP (ru/en — фактические языки пользователей). Расширение — отдельный ADR.
- **Override явным locale от iOS-клиента** — по-прежнему **вне MVP**, [Q-LOCALE-1](../99-open-questions.md#q-locale-1) (`blocks_sprint: none`).

**Требование к зависимостям.** Новых внешних библиотек/SDK не вводится (детект — собственная script-эвристика, чистый Python). Правка `docs/02-tech-stack.md` **не требуется**.

**Требование к схеме БД.** Вводится поле `generation_jobs.content_language` (BCP-47, NOT NULL default `'en'`) + миграция (revises `20260608_0001`). См. [03-data-model → generation_jobs](../03-data-model.md#generation_jobs).

## Alternatives

1. **Сохранить ADR-025 как есть (само-детект Agent 2 + приоритет ответов).** Отклонено — **это и есть источник прод-бага**: недетерминизм LLM-инструкции + каскад через приоритет ответов. Ревизия ровно его и устраняет.

2. **Библиотека язык-детекта (`langdetect`/`py3langid`) → BCP-47 на сервере.** Отклонено для этой задачи: (а) новая внешняя зависимость (правка `02-tech-stack.md`); (б) **шум на коротких промптах** (типичный ввод — одна фраза; библиотеки путают `ru`/`uk`/`bg`, `en`/`de`/`nl`) — это привнесло бы новый класс edge-кейсов ради точности, которая для **директивы** не нужна (финальную точность даёт LLM); (в) детерминизм важнее точности языка — script-подсчёт детерминирован тривиально. **Примечание:** в ADR-025 серверный `langdetect` отклонялся для задачи детекта языка **контента** (где точность важна и Agent 2 всё равно читал весь ввод); здесь задача **проще** (детект **директивы**, финальную точность даёт модель) и детерминизм **критичен** — поэтому серверный детект теперь допустим, но в форме **script-эвристики**, а не библиотеки.

3. **Детект на Agent 1 (LLM само-детект) с прокидкой через поле.** Отклонено: Agent 1 — LLM, само-детект недетерминирован (ровно корень бага). Нужен **серверный** детерминированный детект до LLM.

4. **Программная прокидка по цепочке без поля БД (параметр в сигнатурах + маркер).** Отклонено: не переживает crash-resume между фазами (после рестарта воркер подхватывает по `state` без in-memory параметра); Agent 1 стартует до появления маркера в спеке — нужен персистентный якорь. Поле в БД — единственный crash-устойчивый канал.

5. **Сохранить приоритет ответов над промптом.** Отклонено: именно он порождал каскад (ошибочные англ. вопросы → англ. ответы → англ. сайт). При детерминированном детекте из промпта ответы уже на правильном языке; отдельный сигнал «ответы» избыточен и вреден.
