# ADR-025 — Язык контента сгенерированного сайта: авто-детект из текста пользователя через spec-маркер `**Content language:**`

**Статус:** Accepted
**Дата:** 2026-06-08
**Связано:** [ADR-020](ADR-020-agent-structured-output-tool-use-tolerant-parse-retry.md) (structured-output агентов, канонические output-схемы §I.1a), [pipeline §03 → Язык/локализация](../modules/pipeline/03-architecture.md), [08-product-decisions §Локализация](../08-product-decisions.md), [Q-LOCALE-1](../99-open-questions.md#q-locale-1)

## Context

Прод-баг: сгенерированный сайт выходит на **русском**, хотя пользователь писал промпт и отвечал на уточняющие вопросы на **английском**. Диагностика по коду выявила три корня:

1. **Главный.** Эталонные примеры в системных промтах агентов содержат **кириллицу и локаль-специфичные термины**: `app/pipeline/prompts/agent2_spec_writer.txt` несёт русский пример output (`{"spec_markdown":"# Спецификация\n…"}`) и термин `(ТЗ)`; те же `(ТЗ)` — в `agent4_fixer.txt` и `agent4_editor.txt`. Few-shot-пример на русском смещает язык генерации: Claude копирует язык примера → Agent 2 пишет `spec_markdown` по-русски → Agent 3 строит сайт из русской спеки → сайт русский.
2. Ни один из 4 системных промтов **не инструктирует** про язык user-facing контента.
3. Язык пользователя **нигде не определяется и не фиксируется**; Agent 3 не выставляет корневой `<html lang>`.

Минимальная output-схема Agent 2 в [pipeline §I.1a](../modules/pipeline/03-architecture.md#i1a-канонические-output-схемы-агентов--single-source-of-truth-нормативно) сама использует русский пример (`{"spec_markdown":"# Спецификация\n…"}`) — это часть проблемы (пример в нормативном документе тоже смещает язык).

**Требование пользователя (утверждено, не пересматривается):** язык контента сайта = «язык, на котором отвечает пользователь». Авто-детект из текста пользователя (промпт + ответы на вопросы), **без** изменения публичного API (`POST /projects` body не получает поле `language`), **без** миграции БД, **без** нового параметра в сигнатурах `agentN.py`. Override явным `locale` от клиента — вне MVP.

## Decision

**Авто-детект языка контента один раз Agent 2, фиксация маркером `**Content language:**` в начале `spec_markdown`, прокидывание downstream через уже передаваемый `spec_markdown` (новых колонок/полей/параметров НЕ вводится).**

1. **Детект — Agent 2 (Spec Writer).** Agent 2 — единственный агент, видящий **весь** пользовательский ввод (оригинальный промпт + Q&A-ответы). Он определяет язык контента из этого текста и пишет **весь** user-facing контент спеки на этом языке. `spec_markdown` **обязан** начинаться явной строкой-маркером:

   ```
   **Content language:** <название языка + BCP-47 код>
   ```
   Пример: `**Content language:** English (en)`.

   **Приоритет при расхождении:** язык **ответов** пользователя на уточняющие вопросы важнее языка оригинального промпта (ответы — более свежий и осознанный сигнал; пользователь мог начать промпт на одном языке, но отвечать на другом).

2. **Прокидывание — через `spec_markdown`.** Маркер живёт внутри `spec_markdown` (output Agent 2), который **уже** передаётся downstream (inline `spec_tz` / `spec_ref` в S3). Новый столбец/поле/параметр сигнатуры **не нужен**.

3. **Agent 3 (Builder).** Генерит **весь** видимый контент сайта на языке маркера `**Content language:**` из спеки и выставляет корневой `<html lang="<bcp-47>">`.

4. **Agent 4 (Fixer/Editor).** **Сохраняет** язык контента (не переключает, не переводит). Тот же маркер в той же неизменной спеке — источник языка и для fix-витка, и для edit-витка.

5. **Agent 1 (Interviewer).** Задаёт уточняющие вопросы **на языке оригинального промпта** (пользователь ответит на том же языке) — чтобы Q&A не смешивал языки и детект Agent 2 был чистым.

6. **Дефолт «русский» убирается полностью.** Эталонные примеры в системных промтах **не должны содержать кириллицы / локаль-специфичных строк** (русских примеров output, термина `ТЗ`), чтобы few-shot не смещал язык вывода. Примеры в промтах — нейтральные/англоязычные плейсхолдеры. *(Правка самих `.txt`-промтов — зона backend; здесь фиксируется нормативное требование к их содержимому.)*

**Структура output Agent 2 НЕ меняется** — канонический ключ остаётся `spec_markdown` ([pipeline §I.1a](../modules/pipeline/03-architecture.md#i1a-канонические-output-схемы-агентов--single-source-of-truth-нормативно)); маркер — это первые символы **значения** строкового поля `spec_markdown`, а не новый top-level ключ. Валидатор `_validate_spec` не меняется по сигнатуре; добавляется лишь проверка наличия маркера в начале `spec_markdown` (зона backend, в рамках существующего валидатора).

## Consequences

**Плюсы.**
- Нулевое изменение публичного контракта API и схемы БД — фича целиком в слое промтов + лёгкая валидация маркера.
- Один источник детекта (Agent 2, видит весь ввод) — нет рассинхрона между агентами по языку.
- Маркер машинно-парсим (`**Content language:** <name> (<bcp47>)`) — Agent 3 извлекает BCP-47 для `<html lang>` детерминированно.
- Откат/смена политики не трогает API/БД.

**Минусы / риски.**
- Детект — на стороне LLM (Agent 2), не детерминированный серверный `langdetect`; качество зависит от модели. Принято осознанно (см. Alternatives): серверный детектор — лишняя зависимость и точка рассинхрона, а Agent 2 и так читает весь ввод.
- Смешанный язык ввода (промпт на одном, ответы на другом) разрешается правилом приоритета (ответы > промпт), но edge-кейс «оба языка вперемешку в ответах» отдан на усмотрение модели (приемлемо для MVP).
- Явный locale-override от iOS-клиента (когда пользователь хочет сайт на языке, отличном от языка ввода) — **не покрыт**, вынесен в [Q-LOCALE-1](../99-open-questions.md#q-locale-1) (вне MVP, `blocks_sprint: none`).

**Требование к зависимостям.** Новых внешних библиотек/SDK не вводится (детект — внутри LLM-вызова Agent 2, без серверного `langdetect`). Правка `docs/02-tech-stack.md` не требуется.

## Alternatives

1. **Явный параметр `language`/`locale` в `POST /projects` (и проброс в сигнатуры агентов).** Отклонено: требует изменения публичного API, миграции (хранить выбранный язык), нового параметра в `agentN.py` — прямо против требования пользователя (без API/миграции/сигнатур). Также перекладывает выбор на клиента, тогда как продуктовое решение — «язык = язык пользователя» автоматически.
2. **Серверный детектор языка (`langdetect`/`fasttext` над промптом+ответами в backend) до вызова Agent 2.** Отклонено для MVP: новая внешняя зависимость (правка `02-tech-stack.md`), отдельная точка рассинхрона (серверный детект мог бы разойтись с тем, что «понял» Agent 2 по смыслу), лишний слой. Agent 2 всё равно читает весь ввод — детект внутри него проще и без новой зависимости. Может быть пересмотрено, если LLM-детект окажется ненадёжным (тогда — отдельный ADR + запись в `02-tech-stack.md`).
3. **Детект в Agent 1 (Interviewer) с прокидыванием через отдельное поле.** Отклонено: Agent 1 видит только промпт (без ответов), а приоритет отдан **ответам**; прокидывание потребовало бы нового канала (поля/колонки). Agent 2 видит и промпт, и ответы — естественная единственная точка детекта.
4. **Хранить выбранный язык в `generation_jobs` (новая колонка) и читать downstream из БД.** Отклонено: миграция БД (против требования) и дублирование источника (язык уже выводим из `spec_markdown`-маркера, который и так передаётся). Маркер в спеке — единственный канал.
