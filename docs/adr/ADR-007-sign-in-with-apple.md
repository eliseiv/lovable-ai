# ADR-007 — Sign in with Apple как первичная аутентификация iOS-клиента

**Статус:** Accepted · **Дата:** 2026-06-02

## Context

В Sprint 1 пользователь провижинился вручную с единственным seeded plaintext-ключом (`SEED_API_KEY`), что не масштабируется на multi-user (Sprint 3). Нужен реальный механизм первичной аутентификации iOS-клиента, который:
- не требует от нас хранить пароли пользователей;
- даёт стабильный, уникальный идентификатор пользователя для маппинга на наш `user.id` и далее на Adapty `customer_user_id` (Sprint 3.5);
- естественен для iOS и соответствует требованиям App Store (приложения с авторизацией обязаны предлагать Sign in with Apple, если предлагают сторонние логины).

Варианты: собственная пара логин/пароль; OAuth (Google и др.); **Sign in with Apple** (Apple identity token, верифицируемый по JWKS Apple).

## Decision

**Первичная аутентификация — Sign in with Apple. iOS получает Apple identity token (JWT) и отправляет его на `POST /v1/auth/apple`. Backend верифицирует токен и обменивает его на наш собственный opaque Bearer API-key.**

- **Верификация Apple identity token (server-side):**
  - подпись проверяется по публичным ключам Apple (JWKS `https://appleid.apple.com/auth/keys`, кэшируются с учётом `kid`);
  - `iss == https://appleid.apple.com`;
  - `aud == <наш iOS bundle id / Services ID>` (env `APPLE_AUDIENCE`);
  - `exp` не истёк, `iat`/`nbf` валидны;
  - `nonce` (если передан клиентом) сверяется.
- **Identity:** `sub` из Apple-токена (стабильный per-Apple-team идентификатор пользователя) сохраняется как `users.apple_sub` (UNIQUE). Upsert: найден → существующий user; не найден → создаём нового user + (Sprint 3.5) Adapty `customer_user_id = user.id`.
- **Обмен на наш токен:** backend генерирует собственный opaque API-key (наш Bearer), возвращает его клиенту один раз. Хранится только argon2id-хэш + индексируемый `key_id` ([ADR-008](ADR-008-indexed-api-key-lookup.md)). Apple-токен дальше не используется как Bearer — все последующие запросы идут с нашим ключом.
- **Почему свой токен, а не Apple-токен напрямую:** Apple identity token короткоживущий и не предназначен быть долгоживущим Bearer'ом к нашему API; свой opaque-ключ даёт нам контроль над отзывом, мульти-устройством и временем жизни без зависимости от Apple на каждом запросе.

## Consequences

**Плюсы:** нет хранения паролей; стабильный `apple_sub` для маппинга на Adapty; соответствие App Store; верификация офлайн по JWKS (без вызова Apple на каждый логин, только обновление ключей). Наш opaque-ключ развязывает аутентификацию запросов от Apple.

**Минусы:** зависимость от доступности JWKS Apple при логине (mitigation — кэш ключей); требуется конфигурация `aud` (bundle/Services ID); при «Hide My Email» email недоступен/релейный — поэтому identity строится на `sub`, не на email.

**Зависимость (внешняя, отмечена):** реальные значения `APPLE_AUDIENCE` (bundle id / Services ID) задаются при наличии Apple Developer-конфигурации iOS-приложения. До этого — конфигурируемо через env, тесты используют mock JWKS.

## Alternatives

- **Собственный логин/пароль.** Отвергнут: хранение паролей, восстановление, риск утечки — лишняя ответственность для backend без бизнес-ценности.
- **Apple identity token как постоянный Bearer.** Отвергнут: короткоживущий, не наш контроль отзыва/TTL, зависимость от Apple на каждом запросе.
- **Сторонний OAuth (Google и пр.) первым.** Отложен: целевой клиент — iOS, Sign in with Apple покрывает MVP; добавление провайдеров — отдельное решение позже (не блокирует S3).
