# ADR-003 — Celery vs RQ для фоновых джоб

**Статус:** Accepted · **Дата:** 2026-06-02

## Context

Тяжёлая работа (LLM-вызовы, сборка/деплой) выполняется асинхронно вне API. Требуется:
- две независимо масштабируемые очереди (`queue=llm` по rate-limit Claude, `queue=build` по CPU/изоляции);
- зрелые ретраи с backoff (фейлы Claude, временные ошибки сборки);
- периодический планировщик (sweeper таймаутов уточнений, ресинк Adapty);
- работа на Redis (уже используется как pub/sub и счётчики).

## Decision

**Celery 5.4 на Redis-брокере.** Routing по очередям (`-Q llm`, `-Q build`), `autoretry`/backoff на task, Celery **beat** для периодических задач.

## Consequences

**Плюсы:** namespace-очереди и независимый масштаб из коробки; зрелые retries/backoff; beat для sweeper/ресинка; широкая экосистема и observability.
**Минусы:** тяжелее и сложнее в конфигурации, чем RQ; больше движущихся частей (beat — отдельный процесс).

## Alternatives

- **RQ.** Отвергнут: слабее с множественными очередями и периодическими задачами (нужен сторонний scheduler), менее богатая модель ретраев. Простота RQ не перевешивает потребности проекта в раздельных очередях и beat.
- **Arq / собственный asyncio-воркер.** Отвергнут: меньше зрелости, пришлось бы строить routing/retry/scheduling самим.
