# ADR-024 — Клиентская аутентификация по `user_id` + секрет (register/login без Apple)

**Статус:** Accepted
**Дата:** 2026-06-08
**Связано:** [ADR-007](ADR-007-sign-in-with-apple.md) (Sign in with Apple), [ADR-008](ADR-008-indexed-api-key-lookup.md) (формат Bearer `lv_<key_id>_<secret>`, индексируемый lookup), [ADR-021](ADR-021-admin-plane-and-bonus-credits.md) (login-as / upsert юзера без `apple_sub`).

## Context

Сейчас пользователь создаётся/входит **только** двумя путями:
- `POST /v1/auth/apple` — публичный, требует Apple identity token ([ADR-007](ADR-007-sign-in-with-apple.md));
- `POST /v1/admin/login-as` — операторский, требует `X-Admin-Key` ([ADR-021](ADR-021-admin-plane-and-bonus-credits.md)).

Клиентского пути регистрации/входа **без Apple** и **без админ-ключа** нет. Он нужен для:
1. **Dev/QA на проде** — создать рабочий аккаунт без iOS-устройства и Apple Sign-In;
2. **Кросс-платформа** — не-Apple клиенты (web/Android/CLI), у которых нет Apple identity token;
3. **Перенос/восстановление аккаунта** — вход с другого устройства по сохранённым креденшелам.

Механизм Bearer-токенов (`api_tokens`, `lv_<key_id>_<secret>`, индексируемый O(1) lookup) уже реализован ([ADR-008](ADR-008-indexed-api-key-lookup.md)) и **не меняется** — новый путь лишь добавляет ещё один способ *получить* Bearer-токен, как `/auth/apple`.

## Decision

Добавляем **публичную клиентскую аутентификацию по паре `user_id` + секрет**, сосуществующую с Sign in with Apple. Два новых публичных эндпоинта (rate-limit по IP, как `/auth/apple`):

### 1. `POST /v1/auth/register` — регистрация нового аккаунта
- **Сервер сам генерирует** и `user_id` (`new_user_id()`, формат `u_...`), и секрет (`new_token_secret()`, 256 бит энтропии). **Клиентский `user_id` НЕ принимается** (см. §Security-rationale).
- Создаёт `users` (`apple_sub=NULL`, `adapty_customer_user_id=users.id`, `status='active'`) — как admin-created юзер ([ADR-021 §B](ADR-021-admin-plane-and-bonus-credits.md)).
- Сохраняет **`argon2id(secret)`** в новом поле `users.auth_secret_hash` (сам секрет не хранится/не восстановим).
- Выдаёт Bearer через **существующий** `token_service.issue_token()` (новая строка `api_tokens`).
- Ответ (секрет показывается **ОДИН раз**): `{ user_id, secret, api_key: "lv_...", token_id }`.

### 2. `POST /v1/auth/login` — вход по секрету
- Тело: `{ user_id, secret, device_label? }`.
- **Constant-time argon2-проверка** секрета против `users.auth_secret_hash`.
- Успех → новая строка `api_tokens` через `token_service.issue_token()` → `{ api_key, token_id, user_id }`.
- **ЛЮБОЙ провал** (нет юзера / неверный секрет / у юзера нет секрета / Apple-only-аккаунт) → **ЕДИНЫЙ `401`** без раскрытия причины (как `/auth/apple`), RFC-7807.

### 3. Модель данных: поле на `users`, не отдельная таблица
**Решение:** `users.auth_secret_hash text NULL` — **один секрет на пользователя**, поле прямо на `users`. Отдельная таблица `user_secrets` **отклонена**: проще, один секрет на юзера, не нужна история/мульти-секрет (Bearer-токены и так мульти-устройство живут в `api_tokens`). `auth_secret_hash` — **не** identity-якорь (им остаётся `user_id`/`apple_sub`), поэтому **без UNIQUE**. `NULL` — у Apple-юзеров и admin-created (у них секрета нет). Миграция — `add_column users.auth_secret_hash`, revises текущий head `20260604_0001`.

### 4. Anti-brute-force: IP-лимит + per-user_id лок на /login (РЕШЕНО)
**Решение:** на `/register` и `/login` — **IP-лимит** (переиспользуем `check_login_rate_limit` по IP, как `/auth/apple`). Дополнительно **на `/login` вводится per-`user_id` лок** против перебора секрета **одного** аккаунта.

**Обоснование:** `user_id` **виден в ответах API** (он не секрет — возвращается в `/auth/apple`, `/billing/me`-владельце, листингах). IP-лимит не защищает от распределённого перебора секрета конкретного `user_id` (атакующий знает `user_id` жертвы и крутит секрет с разных IP). 256-битный секрет статистически неперебираем, но **defense-in-depth** обязателен: вводим **per-`user_id` счётчик неудачных попыток** (Redis fixed-window, ключ `rl:login:uid:{user_id}`) — N неудач за окно → временный `429` (`Retry-After`) на этот `user_id` независимо от IP. Параметры — `LOGIN_USER_LOCK_THRESHOLD` (default 10) / `LOGIN_USER_LOCK_WINDOW_S` (default 900). Успешный вход сбрасывает счётчик. Чтобы не превратить лок в user-enumeration-оракул и не раскрыть «такой `user_id` существует», `429` отдаётся **по факту превышения порога попыток для значения `user_id`** независимо от существования юзера (счётчик ведётся по присланному `user_id`-значению). Нормативный контракт — [05-security.md → Клиентская аутентификация по секрету](../05-security.md#клиентская-аутентификация-по-user_id--секрет-adr-024).

### 5. Привязка существующего аккаунта (set/rotate секрета) — РЕШЕНО: добавляем `POST /v1/auth/secret`
Открытый вопрос плана: как Apple-юзер ставит секрет (или secret-аккаунт привязывает Apple) для «переноса/восстановления между платформами». **Решение:** для перекрытия сценария «перенос/восстановление» **MVP включает один Bearer-защищённый эндпоинт `POST /v1/auth/secret`** (set **или** rotate секрета текущего пользователя):
- **Auth:** Bearer (действует за уже вошедшего юзера — через Apple ИЛИ через register/login);
- Сервер генерирует **новый** секрет, пишет `argon2id(secret)` в `users.auth_secret_hash` (set, если был `NULL`; rotate, если был задан — старый секрет инвалидируется, существующие `api_tokens` НЕ отзываются), возвращает `{ user_id, secret }` **один раз**;
- Это закрывает «Apple-юзер ставит секрет» (кросс-платформенный вход тем же аккаунтом) **без** введения отдельной identity-связки Apple↔secret-аккаунта.

**Что осталось вне MVP** (вынесено как [Q-AUTH-1](../99-open-questions.md#q-auth-1), **не блокирует** эту фичу): слияние **двух уже существующих** разных аккаунтов (Apple-аккаунт + отдельно register-созданный) в один — требует account-merge-семантики (перенос проектов/подписок/usage), это отдельное продуктовое решение. До его принятия `register` создаёт **новый** аккаунт; «привязка» = постановка секрета на *свой текущий* аккаунт через `/auth/secret`.

## Consequences

**Положительные:**
- Dev/QA, не-Apple клиенты и восстановление аккаунта получают полноценный путь входа без Apple/админ-ключа.
- Переиспользуются `token_service.issue_token()`, формат `lv_<key_id>_<secret>`, индексируемый lookup, мульти-устройство, revoke — **механизм Bearer не дублируется и не меняется** ([ADR-008](ADR-008-indexed-api-key-lookup.md)).
- Единый `401` и IP-лимит идентичны `/auth/apple` — нет нового класса auth-поведения для клиента.
- `auth_secret_hash` nullable, без UNIQUE → миграция аддитивна, не ломает Apple/admin-юзеров.

**Отрицательные / риски:**
- Появляется **второй носитель долговременного креденшела** на юзера (секрет) дополнительно к Apple. Митигация: argon2id-хэш, секрет только на сервере, никогда не логируется (как `key_hash`), показывается один раз; per-`user_id` лок против перебора.
- Восстановление при утере секрета без Apple — **не покрыто** (нет email/recovery-flow). Для MVP приемлемо: секрет = единственный фактор для secret-only аккаунта, клиент обязан его сохранить (как `api_key`). Recovery-flow — вне scope (Q-AUTH-1).
- `register` без CAPTCHA/email-верификации допускает массовое создание аккаунтов. Митигация: IP rate-limit; quota-gate биллинга всё равно ограничивает ресурсопотребление на аккаунт. Anti-abuse усиление — вне scope MVP.

## Alternatives

1. **Только `user_id` без секрета** — **ОТКЛОНЕНО (решение пользователя):** `user_id` не секретный (виден в ответах API) → любой вошёл бы за чужой аккаунт. Неприемлемо.
2. **Принимать клиентский `user_id` при register** — **ОТКЛОНЕНО:** позволяет захват/коллизию аккаунта (атакующий регистрирует `user_id` жертвы вперёд / угадывает существующий). Сервер генерирует `user_id` сам.
3. **Отдельная таблица `user_secrets`** — **ОТКЛОНЕНО для MVP:** один секрет на юзера не требует строки-на-секрет/истории; поле на `users` проще. Если в будущем понадобятся несколько именованных секретов/история ротаций — вводится отдельным ADR (поле остаётся обратно совместимым: `users.auth_secret_hash` = «текущий» секрет).
4. **Email+пароль** — **ОТКЛОНЕНО:** требует верификации email/recovery-инфраструктуры (SMTP, токены сброса), чего в проекте нет; `user_id`+секрет переиспользует существующий argon2/token-стек и закрывает целевые сценарии (Dev/QA, кросс-платформа, перенос) минимальными средствами.
5. **Только IP-лимит на /login без per-user лока** — **ОТКЛОНЕНО:** не защищает от распределённого перебора секрета известного `user_id` (см. Decision §4). Per-`user_id` лок добавлен как defense-in-depth.
