# ADR-001 — State-machine + диспетчер (task-на-состояние) vs Celery chain

**Статус:** Accepted · **Дата:** 2026-06-02

## Context

Пайплайн генерации проходит через 9 состояний (`CREATED…LIVE`, ветки `FIXING`/`FAILED`) и включает:
- долгую паузу human-in-the-loop (`AWAITING_CLARIFICATION`) — могут пройти минуты/часы до ответа пользователя;
- циклический самовосстанавливающийся подпайплайн (`DEPLOYING→FIXING→BUILDING→DEPLOYING`: фикс → пересборка нового source.tgz → передеплой) с гардами;
- разнородные шаги (LLM-вызовы vs сборка) на разных очередях/хостах.

Нужно, чтобы падение воркера не теряло прогресс и каждый шаг был ретраебельным независимо.

## Decision

Оркестрация — **state-machine, материализованная в колонке `generation_jobs.state`, + диспетчер: отдельный Celery-task на каждое состояние**. Диспетчер по текущему `state` ставит следующий task. Каждый переход пишется в `job_events` (audit + SSE).

- Пауза на `AWAITING_CLARIFICATION` = просто отсутствие задач в очереди (ноль компьюта). Резюм — событийный: `POST /answers` ставит task `SPECCING`.
- Цикл FIXING реализуется повторной постановкой task с инкрементом `retry_count` и проверкой гардов в самом task.
- Crash-resumability: после рестарта диспетчер/sweeper подхватывает джобы по `state`.

## Consequences

**Плюсы:** crash-resumable; пауза без компьюта; пошаговые ретраи; естественный audit-трейл; раздельные очереди для LLM/build.
**Минусы:** больше «движущихся частей», чем у линейного chain; нужен sweeper (beat) для зависших состояний; сам диспетчер — компонент, который надо тестировать (unit на переходы).

## Alternatives

- **Один длинный Celery-task / Celery `chain`** на весь пайплайн. Отвергнут: пауза human-in-the-loop держала бы worker-слот или требовала бы хаков; падение в середине теряет прогресс; цикл FIXING плохо ложится на статичный chain; нельзя ретраить отдельный шаг чисто.
- **Внешний workflow-engine (Temporal/Airflow).** Отвергнут для текущего масштаба как избыточная инфраструктура (принцип простоты, [00-vision NFR Simplicity]).
