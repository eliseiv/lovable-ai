# ADR-005 — No-progress detection через хэш сигнатуры фейла

**Статус:** Accepted
**Дата:** 2026-06-02
**Связано:** Sprint 2 (Fixer loop + resilience), [modules/pipeline/03-architecture.md](../modules/pipeline/03-architecture.md), [Q-COST-1](../99-open-questions.md#q-cost-1)

## Context

Цикл самовосстановления `DEPLOYING → FIXING → BUILDING → DEPLOYING` (фикс → пересборка нового source.tgz → передеплой) ограничен четырьмя гардами (hard cap попыток, cost cap, wall-clock cap, no-progress). Три из них — простые счётчики/таймеры. Четвёртый, **no-progress detection**, должен ловить ситуацию, когда Agent 4 «крутит на месте»: выдаёт патч, который приводит **ровно к тому же фейлу**, что и предыдущая попытка. Без него джоба прожжёт весь `max_fix_attempts`/бюджет на бессмысленные одинаковые итерации, прежде чем упасть.

Нужно детерминированно ответить: «новый фейл — это тот же самый фейл, что и прошлый?» При этом:
- Build-логи **нестабильны буква-в-букву** между прогонами: абсолютные пути (`/var/builds/{job_id}` с разным `job_id`-сегментом запуска), таймстампы, длительности, временные хэши, номера портов, PID. Сравнение сырого лога целиком даст «всегда разные» → no-progress никогда не сработает.
- Сравнение «дерево файлов не изменилось» — недостаточно: Agent 4 почти всегда что-то меняет, но изменение может быть нерелевантным фактической ошибке.

## Decision

Вводим **нормализованную сигнатуру фейла** `failure_signature` — детерминированный хэш (sha256, hex) от **канонизированного** представления причины фейла, а не от сырого лога. Алгоритм канонизации (реализуется в `app/pipeline/failure_signature.py`):

1. Взять класс фейла `failure_class` (машинный, см. ниже) — он всегда входит в сигнатуру.
2. Извлечь из лога **диагностическое ядро**:
   - для build-fail — строки уровня error компилятора/сборщика (`error TS...`, `ERROR in ...`, `Cannot find module ...`, `npm ERR! ...`, exit-code сборки);
   - для health-fail — код/класс ответа health-check (`timeout` / `connection_refused` / `http_5xx` / `http_4xx`) и, если есть, первая строка stderr nginx;
   - для invalid-agent-output — машинный код правила валидации (`path_traversal`, `binary_encoding_mismatch`, `tree_too_large`, ...).
3. **Нормализовать** извлечённое: вырезать/заменить плейсхолдерами нестабильные токены — абсолютные пути → `<PATH>`, таймстампы → `<TS>`, числа длительностей/портов/PID → `<NUM>`, hex-хэши ≥ 8 симв. → `<HEX>`, `{job_id}`-сегмент → `<JOB>`. Привести к нижнему регистру, схлопнуть пробелы, отсортировать множество error-строк (порядок недетерминирован между прогонами).
4. `failure_signature = sha256(failure_class + "\n" + normalized_core)`.

Сигнатура считается **в pipeline** (не в deploy) при получении фейла, до постановки `task_fix`. Текущая сигнатура сравнивается с `generation_jobs.last_failure_signature` (сигнатура **предыдущего** фейла). Если совпала **на ≥ 1 попытке fix подряд** (т.е. патч Agent 4 не сдвинул причину) → гард no-progress срабатывает → `FAILED(no_progress)`. После сравнения `last_failure_signature` перезаписывается текущей.

> Семантика «дважды подряд»: первый build-fail (попытка 0, ещё нет предыдущей сигнатуры) — это вход в цикл, no-progress не применяется. Срабатывает начиная со **второго** фейла: «Agent 4 пропатчил, передеплой дал ту же сигнатуру». Это и есть «та же failing-сигнатура дважды» из ТЗ NFR.

### Уточнение: различение нового failure-event и crash-resume reprocessing (`failure_event_pending`)

Сравнения только сигнатур недостаточно при **crash-resumable** топологии (ADR-001): reconciler (pipeline §E2) после краша воркера переставляет таску по `state=FIXING` и **переобрабатывает тот же** уже-учтённый failure-event. Если бы гард сравнивал только `last_failure_signature == failure_signature`, такой reprocessing совпал бы сам с собой и дал **ложный `no_progress`**.

Поэтому вводится guard-state `generation_jobs.failure_event_pending` (Boolean NOT NULL default false, миграция `20260602_0002`):
- **выставляется в `true`** при рождении **нового** failure-event — на входе в `FIXING` из `DEPLOYING` (`enter_fixing`) и при обработке невалидного патча Agent 4 (виток класса `agent_output_invalid`);
- гард no-progress срабатывает (`FAILED(no_progress)`) **только если** `failure_event_pending = true` **И** сигнатуры совпали — т.е. это **второй distinct failure-event с той же сигнатурой**, а не тот же лог, переобрабатываемый reconciler'ом;
- гард **сбрасывает `failure_event_pending ← false`** при обработке event. Reconciler-перепостановка флаг не выставляет → reprocessing идёт по «не новый event» ветке и no-progress не триггерит.

Нормативный алгоритм (точные ветки) — единственный источник [pipeline §C(d)](../modules/pipeline/03-architecture.md#d-no-progress-detection); здесь — обоснование решения.

## Consequences

- **Плюс:** ловит зацикливание раньше, чем исчерпается hard cap/бюджет → экономия Claude-$ и времени. Детерминированно и тестируемо (unit на фикстурах логов).
- **Плюс:** устойчиво к шуму лога (пути/таймстампы) — нормализация убирает ложные «отличия».
- **Минус / риск:** нормализация неполна — теоретически возможен ложно-положительный «no_progress» (две *разные* ошибки с совпавшим ядром после агрессивной нормализации) или ложно-отрицательный (та же ошибка, но изменился нормализованный текст → сигнатура разошлась). Это **осознанный компромисс**: ложно-положительный обрывает джобу на попытку раньше (пользователь видит последний лог и может пересоздать), ложно-отрицательный просто откладывает обрыв до hard cap. Точная настройка списка нормализаторов под реальные логи — [TD-005](../100-known-tech-debt.md#td-005) (Sprint 6, когда накопятся реальные логи).
- Сигнатура хранится как `text` в `generation_jobs.last_failure_signature` (поле уже есть, Sprint 1).
- Guard-state `generation_jobs.failure_event_pending` (Boolean, миграция `20260602_0002`) корректно разделяет новый distinct failure-event и crash-resume reprocessing → no-progress не даёт ложных срабатываний на reconciler-перепостановках. Оба guard-state-поля (`last_failure_signature`, `failure_event_pending`) — внутреннее состояние гарда, наружу через API не отдаются.

## Alternatives

- **Сравнивать сырой build-лог целиком** — отвергнуто: нестабилен, no-progress никогда не сработает.
- **Сравнивать хэш дерева файлов output Agent 4** — отвергнуто: Agent 4 почти всегда меняет дерево, но не обязательно релевантно ошибке; не отражает «тот же фейл».
- **LLM-судья «это та же ошибка?»** — отвергнуто для S2: лишний Claude-вызов (стоимость, latency, недетерминизм) в гарде, который обязан быть дешёвым и предсказуемым. Возможен как уточнение в Sprint 6.
- **Только hard cap (без no-progress)** — отвергнуто: NFR явно требует no-progress detection; без него runaway-затраты на одинаковых итерациях.
