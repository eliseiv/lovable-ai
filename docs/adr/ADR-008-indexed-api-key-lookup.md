# ADR-008 — Индексируемый lookup API-key (key_id-префикс) вместо O(N) argon2-перебора

**Статус:** Accepted · **Дата:** 2026-06-02

## Context

В Sprint 1 аутентификация (`app/api/dependencies.py`) перебирала всех пользователей и делала `argon2.verify` против каждого `users.api_key_hash` — O(N) по числу юзеров, с дорогим argon2-хэшированием на каждой итерации. На одном seeded-ключе это приемлемо, но при multi-user (Sprint 3) это DoS-вектор и latency-яма. Зафиксировано как [TD-004](../100-known-tech-debt.md#td-004). Дополнительно Sprint 3 вводит мульти-устройство (несколько активных токенов на пользователя), что усугубляет проблему.

Нужен O(1) lookup строки токена с ровно одним constant-time argon2-verify, при этом нельзя восстанавливать сам ключ из БД.

## Decision

**Opaque API-key несёт индексируемый бескриптографический префикс `key_id`; lookup идёт по UNIQUE-индексу `api_tokens.key_id` → ровно одна строка → один constant-time argon2-verify хэша секрета.**

- **Формат ключа, выдаваемого клиенту:** `lv_<key_id>_<secret>`, где:
  - `key_id` — публичный, индексируемый идентификатор токена (`[a-z0-9]{16}`, генерируется случайно при выдаче, не секрет);
  - `secret` — высокоэнтропийная секретная часть (≥ 32 байта энтропии), хранится в БД **только** как argon2id-хэш (`api_tokens.key_hash`).
- **Lookup на каждом запросе:**
  1. распарсить префикс → извлечь `key_id`;
  2. `SELECT ... FROM api_tokens WHERE key_id = :key_id AND revoked_at IS NULL` (UNIQUE-индекс, одна строка);
  3. один `argon2.verify(row.key_hash, secret)` (constant-time);
  4. при успехе — `current_user = row.user_id`, опционально обновить `last_used_at`.
- **`key_id` не секрет:** утечка `key_id` без `secret` бесполезна; argon2 защищает секретную часть. Поиск по `key_id` — обычный B-tree-индекс, не уязвим к timing (constant-time остаётся на argon2-verify секрета).
- **Мульти-устройство:** N строк `api_tokens` на одного `user_id`, каждая со своим `key_id`. Отзыв одного устройства = `revoked_at` одной строки, не трогает остальные.
- **Почему `key_id`-префикс, а не HMAC-индекс по всему ключу:** оба дают O(1); `key_id`-префикс проще (не требует серверного HMAC-секрета и его ротации), self-describing (видно, какую строку искать), и естественно ложится на мульти-устройство (один `key_id` = одно устройство/токен).

## Consequences

**Плюсы:** O(1) lookup независимо от числа юзеров/токенов; ровно один argon2-verify на запрос; снят DoS-вектор TD-004; естественная поддержка мульти-устройства и per-token revoke; сам секрет не восстановим из БД.

**Минусы:** формат ключа клиента меняется (`lv_<key_id>_<secret>`) — несовместим с S1 seeded plaintext-ключом (см. миграционный путь ниже); `key_id` виден в логах/трафике, если не маскировать (mitigation — логировать только `key_id`, никогда `secret`).

**Миграционный путь (совместимость с S1):** `users.api_key_hash` (S1) остаётся как legacy-поле на время миграции; новый код проверяет сначала `api_tokens` (новый путь), при отсутствии префикса `lv_` — fallback на legacy seeded-проверку (single seeded user) до перевода тестов/окружения на новый формат. Полный отказ от legacy-пути — после миграции (отметка в [03-data-model.md](../03-data-model.md#api_tokens-sprint-3) и тестах). Это сохраняет обратную совместимость существующих integration-тестов S1/S2 на время перехода.

## Alternatives

- **O(N) argon2-перебор (S1).** Отвергнут: DoS-вектор и latency, причина TD-004.
- **HMAC-индекс по всему ключу (`api_key_lookup = HMAC(key)`).** Рабочая альтернатива (упоминалась в TD-004), отвергнута как дефолт: требует серверного HMAC-секрета и его ротации, менее естественна для мульти-устройства, чем самоописывающийся `key_id`.
- **Хранить ключ в открытом виде с UNIQUE-индексом.** Отвергнут: компрометация БД = утечка всех ключей; нарушает принцип «в БД только хэш».
